# 测试用例矩阵

下表对应仓库中可运行的 `pytest` 用例和服务测试运行器，重点覆盖奖励发放的幂等、事务边界、接口错误契约、并发库存和数据质量定位。

| 场景 | 前置条件／输入 | 预期结果 |
| --- | --- | --- |
| 正常发奖 | 有效玩家、开启活动、库存充足 | 返回 `201`；库存减 1，玩家余额增加奖励值，写入成功记录 |
| 同请求重试 | 使用相同 `Idempotency-Key` 重复请求 | 返回历史结果并标记 `duplicated: true`；库存与余额不重复变化 |
| 幂等键冲突 | 同一个 Key 对应不同玩家或活动 | 返回 `409`，拒绝写入 |
| 库存不足 | 库存耗尽后发起新请求 | 返回 `409`；余额不再变化 |
| 不存在的资源 | 玩家或活动不存在 | 返回 `404` 及明确错误信息 |
| 缺少幂等键 | 请求头缺少 `Idempotency-Key` | FastAPI 参数校验返回 `422` |
| 空玩家 ID／查询不存在库存 | 空字符串请求体或未知玩家 | 返回 `422`／`404`，错误边界可预期 |
| 冻结账号 | 玩家状态为 `suspended` | 返回 `409`，不写入发奖记录、不扣库存 |
| 领取上限 | 同一玩家达到活动 `per_player_limit` | 返回 `409`，余额和库存不重复变化 |
| 并发扣库存 | 两名玩家同时领取库存为 1 的活动 | 仅一人成功，另一人返回库存不足；库存不出现负数 |
| 重复发奖检查 | 同一玩家、同一活动存在多条成功记录 | `duplicate_reward` 返回失败计数与玩家／活动样本 |
| 异常数据检查 | 构造孤儿记录、非法状态、负余额 | 对应规则返回最多 3 条样本，便于定位 |
| 奖励／库存账实检查 | 奖励金额或当前库存被构造为异常值 | 返回配置金额差异、预期库存与实际库存样本 |
| 服务测试运行记录 | 创建一条 `test_runs` 记录 | 保存 5 个场景的请求、响应、断言、耗时和最终汇总 |
| 测试数据隔离与清理 | 每个服务场景使用 `qa_<run>_*` 前缀创建玩家与活动 | 场景结束后清理临时业务数据，运行结果仍可在 `test_case_results` 回看 |
| Redis/Celery 异步执行 | Docker Compose 中 `EXECUTION_MODE=celery` | API 先返回 `queued`，Worker 从 Redis 取任务并更新运行状态 |

运行命令：

```bash
python3 -m pytest -q --cov=app --cov-report=term-missing
```

所有异常数据均由本地演示脚本构造，不来自外部或真实业务数据。
