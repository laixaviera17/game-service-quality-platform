# 面试讲解提纲

## 业务问题

活动奖励接口很容易因用户重试、网络超时或重复投递产生重复发奖；同时，配置与发奖数据的异常可能在上线后放大为玩家资产问题。

## 我的设计

1. 用 `Idempotency-Key` 识别同一请求重试；同 Key 同请求返回历史结果，同 Key 不同请求拒绝。
2. 用数据库事务在一次操作中检查活动状态/库存、扣减库存、增加玩家余额、写入发奖记录。
3. 将重复发奖、孤儿记录、非法配置、负资产、奖励金额不一致和库存账实不一致拆为独立质量规则；每条规则输出计数和最多 3 条异常样本，便于从看板继续定位数据。
4. 测试运行器会为每个场景生成独立前缀的玩家和活动数据，执行正常发奖、幂等重试、领取上限、冻结账号和并发库存五个场景；每条记录保存请求、响应、断言和耗时，结束后清理临时业务数据。
5. 每次手动检查会把汇总、规则结果和样本写入数据库快照；控制台可查看最近记录或切换到任意一条历史快照。
6. 将玩家或活动不存在映射为 404；库存不足、活动未开启和幂等键冲突映射为 409，并在 API 测试中校验响应契约与余额不变性。
7. 本地开发默认同步执行，Docker Compose 环境启动 MySQL、Redis、API 和 Celery Worker；服务测试任务先进入 Redis，再由 Worker 更新运行状态。这是本地技术演示，不连接真实游戏系统。

## 现场演示顺序

1. 运行 `python3 -m scripts.seed_demo` 和 `python3 -m scripts.run_quality_check`，展示 6 条规则全部通过。
2. 运行 `python3 -m scripts.seed_issue_demo`，再刷新 `/dashboard`，展示 6 条规则的失败计数、严重级别与异常样本。
3. 在控制台点击“执行服务测试”，说明每条场景会留下请求、响应、断言和耗时；随后解释临时数据会清理，结果会保留。
4. 运行 `python3 -m pytest -q --cov=app --cov-report=term-missing`，展示接口、服务层、测试运行器和质量检查用例共同验证核心路径。

## 可继续迭代

- Redis 只承担测试任务队列；下一步可设计独立的幂等状态存储与过期策略，但不能直接把它说成已实现。
- 接入 Allure、JMeter / Locust，补充接口回归、性能基线和压测报告。
- 为失败记录增加趋势统计、样本归档和通知能力。
