# 面试讲解提纲

## 业务问题

活动奖励接口很容易因用户重试、网络超时或重复投递产生重复发奖；同时，配置与发奖数据的异常可能在上线后放大为玩家资产问题。

## 我的设计

1. 用 `Idempotency-Key` 识别同一请求重试；同 Key 同请求返回历史结果，同 Key 不同请求拒绝。
2. 用数据库事务在一次操作中检查活动状态/库存、扣减库存、增加玩家余额、写入发奖记录。
3. 将重复发奖、孤儿记录、非法配置、负资产拆为独立质量规则，统一输出结构化结果给看板或告警系统。

## 可继续迭代

- SQLite 改 MySQL，增加 Redis 幂等缓存与消息队列异步发奖。
- 接入 Allure、JMeter / Locust，做接口回归和压测。
- 为质量规则失败记录增加 LLM 异常归因和测试用例建议，但保留规则引擎作为确定性防线。

## 诚实边界

这是个人模拟项目，不应声称服务过真实游戏、处理过生产玩家数据或已承受真实高并发。
